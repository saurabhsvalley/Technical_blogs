Blog: Creating Dynamic Password History with PAM

Purpose: This blog explains how to create a dynamic password history mechanism with PAM (Pluggable Authentication Modules). PAM currently allows configuring a static number of old passwords to prevent reuse, but does not natively support a dynamic model â€” especially where password changes are restricted to once per day.

What is PAM?

PAM (Pluggable Authentication Module) is a plugin-based authentication system in Linux. It provides a flexible framework to control how users log in and authenticate, without applications having to implement their own authentication logic.

What is the opasswd file?

  • opasswd = Old Passwords file.
  • Used by the pam_unix module when configured with the remember=N option.
  • Purpose: Stores previously used password hashes, preventing users from reusing their last N passwords.

What is the oshadow file?

  • oshadow = Old Shadow file.
  • Acts as a backup of /etc/shadow when changes are made.
  • Purpose: Preserves previous shadow entries, useful for recovery or rollback if corruption occurs during password changes.

ðŸ”¹ Problem Statement

How do we create a dynamic password history in PAM? By default, PAM only supports a fixed remember=N setting. We want a method where the number of previous passwords remembered changes dynamically, based on time elapsed since the last password modification.

ðŸ”¹ Approach 1: Dynamic Password History

Steps

  1. Use the oshadow file to determine the last password modification date for a given user.
  2. When a password is changed:
    • Read the last modification date from oshadow.
    • Calculate the number of days between that date and the current time.
  3. Insert into opasswd a number of password history entries equal to the number of days since the last modification.
  4. Adjust the number of saved password hashes in opasswd accordingly.
  5. Each time a password changes:
    • Backup the current opasswd to opasswd_old.
    • Update both opasswd and opasswd_old with the new password entry.
  6. If the configured number of previous passwords changes (increased or decreased):
    • Reconstruct opasswd from opasswd_old to match the new required count.

Notes:

  • This algorithm works across all PAM versions that use opasswd and oshadow.
  • You can also leverage the chage utility for managing password expiry and aging.
  • Best practice: Always create a copy of opasswd, opasswd_old, and oshadow before making changes. Apply modifications on the copy, then rename it back to the original file for safety.

ðŸ”¹ Pros

  • Password history is tied to days, making it more flexible.
  • Number of previous passwords can be increased or decreased dynamically.

ðŸ”¹ Cons

  • Requires a fast algorithm to efficiently update opasswd.
  • Scalability concerns: For large numbers of users, file size grows quickly.
    Example: With 100 users, each storing 365 passwords (1 year of daily history), the opasswd file will contain 36,500 entries.
